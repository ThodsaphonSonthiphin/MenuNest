💡 Stop and think
claude.ai is about to send you over to Microsoft to log in, and Microsoft must later send you back to claude.ai carrying a pass. But the whole internet can reach Microsoft's login page.

How does Microsoft know (a) which app is asking, (b) that it's safe to send the user back to claude.ai and not an imposter site, and (c) that the thing later claiming the pass is really that app?

Hold that question — the next three steps are the answer, and they're exactly the three things we set up in Azure.
⚠️ This is the exact error you hit — and why
Before we added the 📇 oauth-protected-resource card (Steps 3–4), claude.ai had no way to learn where Microsoft Entra lives or which scope to ask for. So it fell back to a last resort: it tried to register itself as a brand-new app with your sign-in service on the fly (called Dynamic Client Registration).

Microsoft Entra does not let unknown strangers self-register. It refused:
"Couldn't register with MenuNest's sign-in service… add an OAuth Client ID" (ref ofid_bec06ce9…)

That error wasn't really about registration — it was the symptom of a missing instructions card. Serve the card + hand claude.ai a known Client ID, and the whole detour disappears.
📐 Every setting → its one job in the handshake
None of these are busywork. Each one answers a specific question the handshake asks:
📇 protected-resource
metadata (NEW)		"Where do I log in, and what permission do I ask for?" — the card claude.ai reads first. This was the missing piece; without it claude.ai is lost (and tried to self-register → your error).
✅ redirect URI"Is it safe to send the user back here?" — Microsoft only returns the user to addresses on this list. We added claude.ai/api/mcp/auth_callback. Missing it → Microsoft error AADSTS50011.
🔒 client secret"Prove you're really the MenuNest app." — the code came back through the user's browser (snoopable); the secret is the private password claude.ai shows to swap that code for the real token.
🎟️ scope
access_as_user		"What is this app allowed to do?" — names the exact permission, so the token is stamped for your API (aud=e65fd81b) and nothing else.

Trade-off we chose: pointing the card at your specific tenant (d500e2f4-…) keeps the login addresses clean and consistent — but it means only Microsoft accounts that live in your tenant can connect Claude. For you and family added to the tenant, that's fine; arbitrary outside Microsoft accounts could not.
